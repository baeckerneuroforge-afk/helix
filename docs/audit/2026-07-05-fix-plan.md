## F1 — Index für `audit_log.action` (P1)

**Ziel:** `LIKE 'prefix%'`-Filter auf `audit_log.action` werden zu Index-Range-Scans, statt bei jeder Dashboard-Navigation über alle Zeilen im Zeitfenster zu scannen.

**Änderung:**
1. Neue Migration `prisma/migrations/0028_audit_action_index/migration.sql` — analog zu `0014_perf_indexes`:
   ```sql
   -- audit_log.action wird auf den heißen Pfaden per LIKE 'prefix%' gefiltert
   -- (Layout-flag-Badge bei JEDER Navigation, Cockpit, Audit-/Flags-Seite,
   -- Cron-Dedup). text_pattern_ops macht den Prefix-LIKE index-fähig.
   CREATE INDEX "audit_log_org_id_action_created_at_idx"
     ON "audit_log" ("org_id", "action" text_pattern_ops, "created_at" DESC);
   ```
   > Optional: `CREATE INDEX CONCURRENTLY` erwägen, falls die Prod-`audit_log` schon groß ist (vermeidet Table-Lock beim Deploy). `CONCURRENTLY` darf nicht in einer Transaktion laufen — Prisma umschließt Migrationsdateien standardmäßig mit einer Transaktion, daher ggf. als separater manueller Schritt oder via `-- prisma-no-transaction`-Konvention prüfen.
2. `prisma/schema.prisma`: den Index am `AuditLog`-Model deklarieren, damit Schema und DB nicht driften:
   ```prisma
   @@index([orgId, action, createdAt(sort: Desc)], map: "audit_log_org_id_action_created_at_idx")
   ```
   > Hinweis: Prisma kann `text_pattern_ops` im Schema nicht ausdrücken. Die Migration (Quelle der Wahrheit) legt den Operator-Class-Index an; die `@@index`-Deklaration hält Prisma davon ab, einen abweichenden Standard-Index zu erzeugen. Nach `prisma migrate deploy` mit `prisma migrate status` prüfen, dass kein Drift gemeldet wird.

**Ergänzung (optional, gleiche Migration):** Der Raw-SQL-Dedup in `src/lib/loop/tick.ts:53-59` filtert zusätzlich `detail ->> 'metric' = $key`. Für die Metrik-Dedup-Query bringt ein Ausdrucks-Index mehr:
```sql
CREATE INDEX "audit_log_metric_dedup_idx"
  ON "audit_log" ("org_id", (("detail" ->> 'metric')), "created_at" DESC)
  WHERE "action" = 'loop.metric_flagged';
```
Nur aufnehmen, wenn der Loop-Cron messbar langsam ist — der Basis-Index oben deckt den Hauptfall schon.

**Verifikation:**
- `EXPLAIN (ANALYZE, BUFFERS)` auf die Layout-Query (`action LIKE 'flag.%' AND created_at >= …`) vor/nach der Migration → nach: `Index Scan using audit_log_org_id_action_created_at_idx`, keine `Seq Scan`/großer `Rows Removed by Filter`.
- Bestehende Test-Suite grün lassen (`pnpm test`) — es ist ein reiner Index, keine Verhaltensänderung.
- Manuell: Dashboard mit einigen tausend Audit-Zeilen laden, Antwortzeit des Layouts prüfen.

**Risiko/Rollback:** Sehr niedrig. Additiver Index, keine Query-/Verhaltensänderung. Rollback = `DROP INDEX`. Einziger Fallstrick: der `CONCURRENTLY`/Transaktions-Punkt oben.

## F2 — Artefakt-Blobs privat statt öffentlich (S1)

**Ziel:** Generierte Deliverables (Angebote, Rechnungen, Frameworks) sind nicht mehr über eine öffentliche Capability-URL abrufbar, sondern nur über den bereits RLS-gesicherten App-Download-Pfad.

**Kontext, der den Fix klein macht:** Der Download-Pfad reicht die URL nie an den Client. `src/app/api/artifacts/[id]/download/route.ts` prüft die Org (`requireTenant` → `getArtifact` → 404 bei fremd) und **streamt die Bytes selbst** (`getArtifactContent` → `blob.get(blobKey)`). Die öffentliche URL wird von der App also praktisch nicht genutzt — nur `blob.get()` liest sie intern.

**Änderung — nur `src/lib/storage/blob.ts` (+ ggf. eine Test-Anpassung):**
1. In `VercelBlobProvider.put()` (Zeile 64-69): `access: 'public'` → `access: 'private'`. `addRandomSuffix: false` und der `orgId`+UUID-Key bleiben.
2. `VercelBlobProvider.get()` (Zeile 73-82): der aktuelle `fetch(match.url)` funktioniert bei privaten Blobs nicht mehr (die URL ist nicht mehr anonym abrufbar). Auf den token-authentifizierten Zugriff umstellen — je nach `@vercel/blob`-API entweder die `download`/`head`-Funktion mit `token` nutzen oder die per `list({ token })` zurückgegebene autorisierte URL. Vor der Umsetzung die installierte `@vercel/blob`-Version (`2.5.0`) gegen die aktuelle Doc prüfen, welche Signatur privates Lesen unterstützt (vercel:vercel-storage-Skill / Blob-Doku).
3. `VercelBlobProvider.delete()` (Zeile 84-89): nutzt bereits `del(url, { token })` — vermutlich unverändert, aber gegen private Blobs verifizieren.
4. Den Header-Kommentar (Zeile 1-7, „private, no public access") wieder wahr machen — er stimmt dann endlich mit dem Code überein.

**Was sich NICHT ändert:** `FakeBlobProvider` (Tests/Dev, in-memory) bleibt wie er ist. `src/lib/artifacts.ts` und die Download-Route bleiben unverändert — sie arbeiten schon gegen `BlobProvider.get()`, das Bytes liefert.

**Migration bestehender Blobs:** Bereits abgelegte Artefakte sind noch `public`. Optionen: (a) für Pilotdaten akzeptieren, dass alte Blobs public bleiben, neue sind privat; (b) ein einmaliges Skript, das je Artefakt `blobKey` neu mit `access:'private'` hochlädt (Bytes über den alten `get()` holen, `put()` privat, alten löschen). Für den aktuellen Reifegrad ist (a) vertretbar; (b) als Cleanup-Task notieren.

**Verifikation:**
- Neues Artefakt erzeugen; die von Vercel Blob zurückgegebene URL **anonym** (ohne Token, im Inkognito) abrufen → muss **403/nicht abrufbar** sein.
- Denselben Download über `/api/artifacts/[id]/download` als eingeloggter Org-Nutzer → **200 + Bytes**.
- Cross-Tenant: als Org B die Artefakt-ID von Org A abrufen → weiterhin **404** (unverändert, RLS).
- `pnpm test` (Fake-Provider) grün; die Artefakt-Tests dürfen sich nicht ändern.

**Risiko/Rollback:** Mittel — die Read-Semantik privater Blobs hängt an der exakten `@vercel/blob`-API. Fehler äußern sich sofort als fehlschlagender Download-Test (fail-closed, kein stiller Leak). Rollback = `access` zurück auf `'public'` und `get()` zurück auf `fetch(url)`. **Kein DB-/Schema-Change**, daher sauber reversibel.

## F3 — Race im Auto-Correction-Tageslimit schließen (S2)

**Ziel:** Die harte Grenze `MAX_AUTO_CORRECTIONS_PER_DAY = 3` pro Org/24h hält auch unter gleichzeitig entstehenden Kriterien-Flags — kein Über-Starten mehr.

**Ursache (verifiziert):** In `src/lib/loop/auto-correct.ts` liest `maybeAutoCorrect` den Zähler (`autoCorrectionsInWindow`) in Tx #1 und startet den Run in einem **separaten** Aufruf (`startCorrectionRun`), der die zählende Audit-Zeile `loop.auto_correction_started` erst **danach**, in `src/lib/loop/correct.ts:163-177`, in **wieder** einer eigenen Tx schreibt. Der Increment läuft der Entscheidung hinterher → zwei parallele Trigger lesen beide `used = 2 < 3` und starten beide. Der Code-Kommentar (`auto-correct.ts:96-99`) benennt die Richtung falsch: der reale Fehler ist **Over-Start**, nicht Under-Start. Es existiert bislang **kein** Lock-Primitive im Repo, daher führt dieser Fix ein neues, eng begrenztes Muster ein.

**Änderung — bevorzugte Variante (Advisory-Lock, minimal-invasiv):**
1. In `maybeAutoCorrect` als **erste** Anweisung der Entscheidungs-Transaktion einen per-Org-Transaktions-Advisory-Lock nehmen — er serialisiert konkurrierende Auto-Correction-Entscheidungen *desselben* Tenants, ohne globale Isolationsstufe oder Schema-Änderung. Weil `withTenant` ohnehin genau einen Backend pinnt, greift der `xact`-Lock sauber und wird bei COMMIT/ROLLBACK automatisch freigegeben:
   ```ts
   await tx.$queryRaw`SELECT pg_advisory_xact_lock(hashtext(${'loop-autocorrect:' + orgId}))`;
   ```
2. **Entscheidung und Start in dieselbe Transaktion falten.** Solange der Start (`startCorrectionRun` → `startRun`) in einer separaten Tx läuft, schützt der Lock nichts. Zwei gangbare Wege:
   - **(a) Zählen gegen den Start-Zeitpunkt statt gegen die Audit-Zeile:** eine dedizierte, tenant-scoped Zähl-Zeile bzw. das Zählen bereits **gestarteter** `skill_runs` mit `isCorrection = true` im Fenster (die Zeile wird *at creation* in `startRun` geschrieben — also *vor* dem Audit, kein Hinterherlaufen). Dann kann der bestehende Ablauf bleiben, und der Lock + das Zählen der `is_correction`-Runs im selben Tx genügt. Das ist wahrscheinlich die sauberste Änderung, weil `skill_runs.is_correction` schon existiert (aus Schritt E) und at-creation gesetzt wird.
   - **(b)** Alternativ eine explizite Zähler-Zeile pro (Org, Tag) mit atomarem `INSERT … ON CONFLICT … DO UPDATE SET n = n+1 WHERE n < limit RETURNING` — überschreitet nie, braucht aber ein neues Schema.
   > Empfehlung: **(a)** — kein Schema-Change, nutzt den bereits vorhandenen strukturellen Marker, und der Zähl-Zeitpunkt (Run-Erzeugung) liegt korrekt *vor* der Start-Entscheidung des nächsten Triggers.
3. Den irreführenden Kommentar (`auto-correct.ts:96-99`) korrigieren: klarstellen, dass ohne Lock **über**-gestartet würde, und dass der Advisory-Lock + at-creation-Zählung das verhindert.

**Was gleich bleibt:** Die vier Bremsen als Konzept, das Approval-Gate (`startCorrectionRun` ruft nie `approve()`), der Anti-Loop-Marker (`is_correction`), und dass `maybeAutoCorrect` best-effort/never-throw nach dem Flag-Commit läuft (`evaluate.ts:149`).

**Verifikation:**
- **Neuer Test** `tests/loop-auto-correct-race.test.ts`: an einem `autonomous`-Tenant mit `used = 2` **mehrere** `maybeAutoCorrect`-Aufrufe *gleichzeitig* auslösen (`Promise.all`) und asserten, dass **genau einer** startet und die übrigen `limit_reached` liefern — die Kernzusage. Ohne den Fix schlägt der Test fehl (mehrere Starts).
- Bestehende `tests/loop-auto-correct.test.ts` / `loop-autonomy.test.ts` müssen grün bleiben (sequenzielles Verhalten unverändert).

**Risiko/Rollback:** Mittel. Der Advisory-Lock ist per-Org und transaktions-lokal — keine globale Kontention, keine Deadlock-Fläche (nur ein Lock je Entscheidung). Falls Variante (a) das Zähl-Verhalten subtil ändert, fängt der neue Race-Test es. Rollback = Lock + Zähl-Umstellung entfernen. Kein Schema-Change bei Variante (a).

## F4 — Serverless-Connection-Pool konfigurieren (P2)

**Ziel:** Auf Vercel keine Connection-Exhaustion mehr, wenn viele Function-Instanzen gleichzeitig interaktive `withTenant`-Transaktionen öffnen.

**Kontext:** `withTenant` ist unter Transaction-Mode-Pooling bereits korrekt (der GUC ist `set_config(..., is_local:=true)`, transaktions-scoped — im Code dokumentiert). Es fehlt rein die operative Konfiguration; **kein Code-Change am Isolationsmechanismus nötig.**

**Änderung (Config/Infra, keine Logik):**
1. **Gepoolten Endpoint für die Runtime nutzen.** Auf Neon: den **pooled** Connection-String (PgBouncer, Transaction-Mode) als `DATABASE_URL` setzen und `?pgbouncer=true` anhängen (schaltet Prismas benannte Prepared Statements ab, die Transaction-Pooling nicht halten kann). Der Neon↔Vercel-Integration-Wert dafür ist der gepoolte Host; der **ungepoolte** bleibt `DIRECT_DATABASE_URL` für Migrationen (so wie `scripts/vercel-build.sh` es bereits erwartet).
2. **`connection_limit` je Instanz kappen**, damit `(Instanzen × Pool)` nicht `max_connections` sprengt — an `DATABASE_URL` anhängen, z. B. `&connection_limit=1` (bei Serverless + externem Pooler ist ein kleiner Wert üblich; genauen Wert an Neon-Plan/`max_connections` ausrichten). Ggf. zusätzlich `&pool_timeout=…`.
3. `.env.example` entsprechend dokumentieren (der Abschnitt erwähnt PgBouncer schon hypothetisch — auf „in Prod so setzen" konkretisieren) und die zwei fehlenden Variablen `CRON_SECRET`, `BLOB_READ_WRITE_TOKEN` ergänzen (kleine Doku-Lücke aus dem Audit).

**Reihenfolge-Kopplung:** F5 (Embedding aus der Tx ziehen) verstärkt diesen Fix, weil dann Transaktionen nicht mehr über Netz-Latenz offen gehalten werden. F4 wirkt aber auch allein.

**Verifikation:**
- Nach dem Umstellen einen Lasttest / mehrere parallele Skill-Runs gegen das Preview/Prod-Deployment fahren und die aktiven Verbindungen beobachten (`SELECT count(*) FROM pg_stat_activity WHERE usename='app_user'`) — muss unter dem erwarteten Ceiling bleiben.
- Smoke-Test, dass die App mit `pgbouncer=true` weiter korrekt läuft (Prisma-Queries, `withTenant`) — die Isolations-Tests decken die Korrektheit unter Pooling konzeptionell schon ab; hier geht es um Betrieb.
- Sicherstellen, dass Migrationen weiter über `DIRECT_DATABASE_URL` (ungepoolt) laufen.

**Risiko/Rollback:** Niedrig-Mittel und rein konfigurativ. Häufigste Falle: `pgbouncer=true` vergessen → Prepared-Statement-Fehler unter Last (laut, nicht still). Rollback = alte `DATABASE_URL` zurücksetzen. Kein Code-, kein Schema-Change.

## F6 — Deploy-Zeit-Assertion für `app_user`-Rollenattribute (S3)

**Ziel:** Die tragende Annahme „`app_user` ist `NOBYPASSRLS`, kein Superuser, kein Table-Owner" wird bei jedem Deploy **verifiziert**, statt nur in einem Local/Docker-Init-Skript und einem `.env.example`-Kommentar zu leben.

**Änderung — zwei Optionen (eine genügt; (a) bevorzugt):**
- **(a) Migration als Guard** — neue Migration `0029_assert_app_user_hardening/migration.sql`, die fehlschlägt, wenn die Invariante verletzt ist. Läuft als Owner (`DIRECT_DATABASE_URL`), prüft aber die *App*-Rolle:
  ```sql
  DO $$
  DECLARE r record;
  BEGIN
    SELECT rolsuper, rolbypassrls INTO r FROM pg_roles WHERE rolname = 'app_user';
    IF NOT FOUND THEN RAISE EXCEPTION 'app_user role missing'; END IF;
    IF r.rolsuper OR r.rolbypassrls THEN
      RAISE EXCEPTION 'app_user must be NOSUPERUSER and NOBYPASSRLS (got super=%, bypassrls=%)', r.rolsuper, r.rolbypassrls;
    END IF;
    IF EXISTS (
      SELECT 1 FROM pg_tables
      WHERE schemaname='public' AND tableowner='app_user'
    ) THEN
      RAISE EXCEPTION 'app_user must not own any public table (FORCE RLS would still apply, but this is a misconfiguration)';
    END IF;
  END $$;
  ```
  Vorteil: läuft automatisch bei `prisma migrate deploy` im `vercel-build.sh`; ein falsch provisioniertes Neon-Setup **bricht den Deploy**, statt still RLS zu unterlaufen. Nachteil: Assertion nur zum Migrations-Zeitpunkt (nicht bei jedem Request) — für eine Rollen-Eigenschaft, die sich praktisch nie ändert, ausreichend.
- **(b) Startup-Check** — die identische Prüfung, die `tests/isolation.test.ts:52-64` schon macht (`current_user`, `rolsuper`, `rolbypassrls`), in `src/instrumentation.ts` beim App-Start ausführen und bei Verletzung laut fehlschlagen/alarmieren. Ergänzt (a) um Laufzeit-Abdeckung, kostet einen Query beim Boot.

> Der Test in `tests/isolation.test.ts` beweist diese Eigenschaften bereits in CI. F6 zieht dieselbe Zusage in **Prod-Deploy/Boot**, wo CI nicht hinreicht.

**Ergänzung (Doku, kein Code):** Im Threat-Model/README festhalten: in Local/Docker/CI ist der Owner (`ergane`) Superuser (umgeht auch FORCE RLS) — das ist nur tragbar, weil die App ausschließlich als `app_user` verbindet und die Owner-URL rein für DDL dient. Owner-Credential nie in die App-Runtime-Umgebung.

**Verifikation:**
- Migration/Startup-Check gegen ein absichtlich falsch konfiguriertes Rollen-Setup laufen lassen (lokal eine Rolle mit `BYPASSRLS` bauen) → muss **fehlschlagen**.
- Gegen das korrekte Setup → **grün**, Deploy läuft durch.

**Risiko/Rollback:** Niedrig. Reiner Guard, ändert kein Verhalten im Normalfall. Einziges Risiko: ein legitimer, aber abweichender Prod-Rollenname → dann die Prüfung parametrisieren. Rollback = Migration/Check entfernen.

## F7 — Header, CRON-Vergleich, Deps, Logging (S4, S5, S6-Teil, D1, Kleinkram)

Fail-closed-bei-fehlendem-Secret (503) beibehalten.
4. **`next`-Manifest-Floor anheben (D1).** In `package.json` `next: ^15.1.6` → `^15.5.19` (oder mind. `^15.2.3`), damit die CVE-2025-29927-sichere Untergrenze nicht nur vom aktuellen Lockfile + Clerk-Peer abhängt. `pnpm install` re-run, Lockfile committen.
5. **`postcss`-CVE (D1).** `pnpm update` bzw. Next-Patch-Bump, sodass das transitive `postcss` auf `>=8.5.10` geht. `pnpm audit --prod` danach → 0 moderate für `postcss`. (Nicht direkt ausnutzbar hier, daher niedrige Dringlichkeit; im selben PR aber billig mitzunehmen.)
6. **`console.*`-Masking-Lücken (S6-Teil).** Die drei Stellen (`settings/actions.ts:350`, `knowledge/actions.ts:123` und `:170`) von rohem `console.info/error` auf `logInfo`/`logError` umstellen, damit sie durch `maskSecrets` laufen. Keine Inhalte/PII heute betroffen — Defense-in-Depth.

> **Nicht** in F7: der substanzielle Teil von **S6** (Prompt-Injection-Härtung des Framework-Skills — Delimiter-Neutralisierung, Vorarbeits-Feedback als untrusted). Das ist Content-Integrität, kein Privilege-Bypass, und eine eigene, sorgfältigere Änderung wert. Als separates, optionales Paket **F8** vormerken, nicht mit dem Kleinkram vermischen.

**Verifikation:**
- Response-Header eines Prod/Preview-Requests prüfen: `Strict-Transport-Security` vorhanden; `Content-Security-Policy` (nicht `-Report-Only`), sobald enforced.
- Cron-Routen mit falschem/fehlendem Bearer → weiter 401/503; mit korrektem → 200.
- `pnpm audit --prod` → keine moderate/high; `pnpm test` grün.

**Risiko/Rollback:** Niedrig, jede Teiländerung einzeln reversibel. Einzige Betriebs-Vorsicht: CSP erst nach Report-Only-Beobachtung enforcen, damit nichts unerwartet bricht.
